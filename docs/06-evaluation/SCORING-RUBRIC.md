# Evaluation Scoring Rubric

> 6단계: 상세 채점 기준
> 조직의 특성과 문화에 맞는 자체적인 평가 프레임워크

---

## Overview

평가는 4가지 기준에 대해 0~5점으로 채점한다.
각 기준은 가중치가 다르며, 가중 평균 **3.2/5.0 이상**이 PASS.

```
Weighted Score = (Relevance × 0.25) + (Faithfulness × 0.30) + (Correctness × 0.25) + (Quality × 0.20)
```

> **Faithfulness(충실도)가 가장 높은 가중치(30%)**인 이유:
> 하네스 규칙 준수가 멱등성의 핵심이기 때문. 아무리 잘 동작해도 하네스를 무시하면 일관성이 깨진다.

---

## Criterion 1: Relevance (연관성) — Weight 0.25

> 요구사항을 정확히 충족하는가?

| Score | Description | Example |
|-------|-------------|---------|
| **5** | 요구사항을 완벽히 충족. 불필요한 코드 0줄 | "사용자 목록 API" 요청에 정확히 해당 엔드포인트만 구현 |
| **4** | 거의 충족. 1-2줄의 불필요 코드 | 요청한 기능 + 불필요한 console.log 1개 |
| **3** | 대체로 충족. 약간의 범위 이탈 | 요청한 기능 + 요청하지 않은 정렬 기능 추가 |
| **2** | 부분적 충족. 주요 기능 누락 또는 과잉 | 사용자 목록 요청에 CRUD 전체 구현 |
| **1** | 거의 무관한 결과물 | 사용자 목록 요청에 인증 시스템 구현 |
| **0** | 완전히 관련 없음 | |

---

## Criterion 2: Faithfulness (충실도) — Weight 0.30

> CLAUDE.md의 모든 하네스 규칙을 준수하는가?

| Score | Description | Example |
|-------|-------------|---------|
| **5** | 모든 규칙 완벽 준수 | 네이밍, 임포트, 멱등성, 패턴 모두 준수 |
| **4** | 1개 미만 경미한 위반 | 임포트 순서가 1곳에서 어긋남 |
| **3** | 1-2개 경미한 위반 | 파일명 1개 camelCase + 임포트 순서 1곳 |
| **2** | 3개+ 위반 또는 구조적 규칙 위반 | default export 사용 + 인라인 함수 다수 |
| **1** | 다수의 규칙 위반 | any 타입 다수 + 네이밍 불일치 + 패턴 무시 |
| **0** | 하네스를 완전히 무시 | CLAUDE.md를 읽지 않은 듯한 결과물 |

### Violation Severity

| Severity | Rule Category | Impact |
|----------|-------------|--------|
| **Critical** | 멱등성 규칙 위반 (Math.random, 인라인 함수 등) | -2점 |
| **Major** | 구조 규칙 위반 (default export, 디렉토리 구조) | -1.5점 |
| **Minor** | 네이밍/포맷 위반 (자동 수정 가능) | -0.5점 |

---

## Criterion 3: Correctness (정확성) — Weight 0.25

> 코드가 정확하고 안전한가?

| Score | Description | Example |
|-------|-------------|---------|
| **5** | 컴파일 깨끗. 모든 엣지 케이스 처리 | null check, 빈 배열, 에러 핸들링 완벽 |
| **4** | 컴파일 깨끗. 주요 엣지 케이스 처리 | 기본 에러 핸들링 O, 극단 케이스 1-2개 누락 |
| **3** | 컴파일 가능. 일반 케이스 처리 | Happy path 동작, 에러 시 미처리 |
| **2** | 컴파일 경고 또는 논리 버그 | TypeScript 경고, off-by-one 에러 |
| **1** | 컴파일 에러 | import 누락, 타입 불일치 |
| **0** | 근본적으로 작동 불가 | |

---

## Criterion 4: Source Quality (소스 품질) — Weight 0.20

> 인간이 읽고 유지보수하기 좋은가? (휴먼 리더블)

| Score | Description | Example |
|-------|-------------|---------|
| **5** | 모범적. 명확한 네이밍, 적절한 추상화, 중복 0 | 코드만 봐도 의도가 명확 |
| **4** | 우수. 읽기 쉽고 구조 깔끔 | 대부분 명확하나 변수명 1-2개 개선 여지 |
| **3** | 보통. 읽을 수 있으나 정리 필요 | 동작하지만 함수가 길거나 중복 약간 |
| **2** | 미흡. 가독성 떨어짐 | 중첩 조건문, 긴 함수, 불명확한 변수명 |
| **1** | 불량. 이해 어려움 | 스파게티 코드, 매직 넘버 다수 |
| **0** | 판독 불가 | |

## Scoring Example

```
Task: "사용자 프로필 카드 컴포넌트 구현"

| Criterion      | Score | Weight | Weighted |
|----------------|-------|--------|----------|
| Relevance      | 5/5   | 0.25   | 1.25     |
| Faithfulness   | 4/5   | 0.30   | 1.20     |
| Correctness    | 4/5   | 0.25   | 1.00     |
| Source Quality  | 5/5   | 0.20   | 1.00     |
| **Total**      |       |        | **4.45** |

Verdict: PASS (4.45 >= 3.2)
```

## Organizational Customization

> 영상 [36:38]에서 강조: "범용 지표가 아닌, 조직 문화와 비즈니스 목적에 최적화된 평가 프레임워크를 구축해야 한다"

이 루브릭은 **기본 템플릿**이다. 조직에 맞게 아래를 조정:

1. **가중치 조정**: 보안이 중요한 조직은 Correctness 가중치 상향
2. **기준 추가**: 성능, 접근성, 국제화 등 조직별 기준 추가 가능
3. **임계값 조정**: 팀 역량에 따라 3.2를 3.5 또는 3.0으로 조정
4. **허용 오차 정의**: 어떤 종류의 오류가 허용되고 안 되는지 명시
